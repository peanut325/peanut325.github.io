Git

概述

Git 是一个免费的、开源的分布式版本控制系统，可以快速高效地处理从小型到大型的各种项目。

Git 易于学习，占地面积小，性能极快。 它具有廉价的本地库，方便的暂存区域和多个工作流分支等特性。其性能优于 Subversion、CVS、Perforce 和 ClearCase 等版本控制工具。

版本控制

  • 版本控制是一种记录文件内容变化，以便将来查阅特定版本修订情况的系统。
  • 版本控制其实最重要的是可以记录文件修改历史记录，从而让用户能够查看历史版本，方便版本切换。

集中式版本控制工具

  • CVS、SVN(Subversion)、VSS等是集中式版本控制，都有一个单一的集中管理的服务器，保存

    所有文件的修订版本，而协同工作的人们都通过客户端连到这台服务器，取出最新的文件或

    者提交更新。

  • 弊端：在服务器宕机后，所有人都无法更新和提交，也无法协同工作。

分布式版本控制工具

  • Git、Mercurial、Bazaar、Darcs等是分布式版本控制工具，客户端提取的不是最新版本的文件快照，而是把代码仓库完整地镜像下来（本地库）。一旦有宕机故障，事后都可以通过其他客户端的本地仓库进行恢复。

  • 优点：

    • 服务器断网的情况下也可以进行开发（因为版本控制是在本地进行的），与集中式最大区别就是可以本地进行版本控制。

    • 每个客户端保存的也都是整个完整的项目（包含历史记录，更加安全）

Git工作机制

代码上传到本地库之后，就是以版本号来控制，在工作区和暂存区可以删除的数据，而在本地库中就不能删除上一个版本数据。

Git和托管中心

代码托管中心是基于网络服务器的远程代码仓库，一般我们简单称为远程库。

  • 局域网：GitLab
  • 互联网：GitHub ， Gitee（码云）

Git安装

安装教程

桌面单击右键 -> 点击Git Bush Here 命令行工具 -> 输入git –version可以查看版本号，同时在Windows命令行也可以查看

Git常用命令

设置用户签名

基本语法

  • git config –global user.name 用户名
  • git config –global user.email 邮箱

查看目录下的.gitconfig文件可以查看设置的全局签名。

作用

  • 签名的作用是区分不同操作者身份。用户的签名信息在每一个版本的提交信息中能够看到，以此确认本次提交是谁做的。Git 首次安装必须设置一下用户签名，否则无法提交代码。

  • 这里设置用户签名和将来登录 GitHub（或其他代码托管中心）的账号没有任何关系。

初始化本地库

基本语法

  • git init

    在目录下使用初始化命令后，会生成 .git 的隐藏文件，表示该文件是git管理的文件。

查看本地库状态

基本语法

  • git status

    • 首次查看，没有文件

    • 加入文件后，再次查看，会检测到未追踪的文件

添加至暂存区

基本语法

  • git add 文件名

    通过查看本地库状态，可以检测到暂存区有文件。

提交至本地库

基本语法

  • git commit -m “日志信息” 文件名

    • 提交文件后，查看本地库状态，显示没有文件需要提交。

    • 此时修改文件，检测到有文件被修改

    • 需要重新提交后，才会检测到没有文件需要提交。

查看日志

基本语法

  • git reflog

    可以查看我们提交文件的版本号（黄字部分）。

  • git log（查看详细日志） 可以查看提交用户

版本穿梭

基本语法

  • git reset –hard 版本号

    此时指针指向的则是你切换的版本，文件中的内容也会自动切换为以前的内容。

Git分支

介绍

在版本控制过程中，同时推进多个任务，为每个任务，我们就可以创建每个任务的单独

分支。使用分支意味着程序员可以把自己的工作从开发主线上分离开来，开发自己分支的时

候，不会影响主线分支的运行。对于初学者而言，分支可以简单理解为副本，一个分支就是

一个单独的副本。（分支底层其实也是指针的引用）

不同的功能可以创建分支修改，在进行合并。

好处

  • 同时并行推进多个功能开发，提高开发效率。
  • 各个分支在开发过程中，如果某一个分支开发失败，不会对其他分支有任何影响。失败的分支删除重新开始即可。

分支基本操作

查看分支

基本语法

  • git branch -v

    此时只有一个master分支，且指针指向master分支

创建分支

基本语法

  • git branch 分支名

    此时还是在master分支上，修改master分支上的内容，hot-fix上的分支将不会被修改。

切换分支

基本语法

  • git checkout 分支名

    • 此时指针切换指向的分支

    此时已经切换到hot-fix分支上。

    • 此时修改hot-fix上的文件，并提交到本地库，master上并没有出现修改过的数据

    • 如果在其他分支修改了但未提交到本地库，master会同步工作区做的修改。

合并分支

基本语法

  • 切换到master git merge 分支名

产生冲突

  • 冲突产生的表现：后面状态为 MERGING
  • 冲突产生的原因：合并分支时，两个分支在同一个文件的同一个位置有两套完全不同的修改。Git 无法替我们决定使用哪一个。必须人为决定新代码内容。

解决冲突

  • 产生冲突后，编辑有冲突的文件，删除特殊符号，决定要使用的内容（特殊符号：<<<<<<< HEAD 当前分支的代码 ======= 合并过来的代码 >>>>>>> hot-fix）

  • 此时再合并，注意此时使用 git commit 命令时不能带文件名

    注意后边的MERGING消失。

Github

GitHub是一个面向开源及私有软件项目的托管平台，因为只支持Git作为唯一的版本库格式进行托管，故名GitHub。

创建仓库

创建账户后，创建仓库即可，仓库名一般和本地库名字一致。

远程仓库常用命令

创建远程仓库别名

基本语法

  • git remote -v 查看当前所有远程地址别名

  • git remote add 别名 远程地址 （远程地址为创建好仓库后的生成的https协议地址）

推送本地分支到远程仓库

基本语法

  • git push 别名 分支 （中途需要登录）

    中途需要登录Github服务器在国外，可能会有超时问题，失败后需要重复推送。

克隆远程仓库到本地

基本语法

  • git clone 远程地址

    此时进入克隆出的目录查看，会自动创建别名。

克隆结果：

  • 拉取代码
  • 初始化本地库
  • 创建别名

拉取远程库内容

基本语法

  • git pull 远程库地址别名 远程分支名

团队内协作

  • 需要邀请成员，成员才可以push

  • 填入想要邀请的成员

  • 复制地址并发送给邀请的用户

  • 被邀请的用户可以在这个账号中的地址栏复制收到邀请的链接，点击接受邀请。

现在团队内成员即可push代码到远程库中。

跨团队协作

  • 将远程仓库的地址复制发给邀请跨团队协作的人。

  • 在被邀请人的GitHub 账号里的地址栏复制收到的链接，然后点击 Fork 将项目叉到自己的本地仓库。

  • 编辑后，可以填写提交的描述信息，并提交

  • 此时点击pull，创建新的请求。

  • 创建请求后，在邀请人的Github账号会接收到一个Pull request请求。

    此时可以进入聊天室讨论。

  • 邀请人可以点击代码查看修改的地方是否正确，如果无误，可以点击Merge pull request合并代码。

IDEA集成Git

忽略文件

  • 在IDEA生成项目时会生成一些特定文件，与项目的实际功能无关，不参与服务器上部署运行。把它们忽略掉能够屏蔽 IDE 工具之

间的差异。

忽略方法

方法一：

直接配置

  • 创建忽略规则文件 xxxx.ignore（前缀名随便起，建议是 git.ignore），这个文件的存放位置原则上在哪里都可以，为了便于让~/.gitconfig 文件引用，建议也放在用户家目录下

  • git.ignore 文件模版内容

  • 在.gitconfig 文件中引用忽略配置文件（此文件在 Windows 的家目录中）

    注意：这里要使用“正斜线（/）”，不要使用“反斜线（\）”

方法二：

使用IDEA中的 .ignore 插件，方便操作。

  • 在插件市场安装插件

    安装完成后记得重启IDEA。

  • 创建模板

    填写号name后，将模板复制到template区域

  • 右键单击项目 -> new -> 创建新的.ignore文件

  • 此时可以生成选择我们创建的模板

  • 生成后忽略的文件夹变为灰色

  • 也可以手动将文件忽略，右键单击

  • 也可以使用token登录，需要在GitHub中生成token

分享工程

  • 分享目前的工程分支在github上，此时会在你的账户下创建一个新的仓库

Push到远程库

  • 进入到push窗口

  • 在push界面中，可以选择你要push的仓库

  • 可以自己创建上传的仓库 Define remote

    name可以自己起名，URL为仓库的地址，可以使用https和ssh（推荐）

  • 可以管理remote连接

注意：

push 是将本地库代码推送到远程库，如果本地库代码跟远程库代码版本不一致，push 的操作是会被拒绝的。也就是说，要想 push 成功，一定要保证本地库的版本要比远程库的版本高！因此一个成熟的程序员在动手改本地代码之前，一定会先检查下远程库跟本地代码的区别！如果本地的代码版本已经落后，切记要先 pull 拉取一下远程库的代码，将本地代码更新到最新以后，然后再修改，提交，推送！

注意:

pull 是拉取远端仓库代码到本地，如果远程库代码和本地库代码不一致，会自动合并，如果自动合并失败，还会涉及到手动解决冲突的问题。
